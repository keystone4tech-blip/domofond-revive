## Дата: 2026-05-31 (Интеллектуальный Fuzzy-автокомплит + Двухэтапный выбор Улица/Дом + Нечеткий Dice-поиск по биграммам + SQL представление unique_houses)
### Изменения:
- **Двухэтапный выбор (Улица -> Дом)**: Переработан UX ввода адреса. Убран обязательный ввод города — теперь поле озаглавлено как «Улица и дом». При вводе сначала выводятся уникальные чистые улицы (без городов и домов). После выбора улицы автоматически подставляется `, д. ` и выводятся только доступные номера домов на этой улице. Город сохраняется в эталонном формате в фоновом режиме для корректной верификации и сопоставления лицевых счетов.
- **Нечеткий поиск (Fuzzy Search) по биграммам**: Реализован локальный Dice-коэффициент сходства биграмм (Dice Coefficient) на фронтенде. Поиск устойчив к опечаткам (например, ввод «джигитайская» успешно находит «Бжегокайская (ул)», а «девятая тихая» находит «9-я Тихая (ул)»).
- **SQL представление `unique_houses`**: В PostgreSQL СУБД создана вьюха `unique_houses` для группировки 4700+ записей из `accounts` до 70 уникальных адресов домов. Это позволило делать мгновенный поиск без переполнения `LIMIT 150` и полностью исключило дублирование подъездов в результатах.
- **Локальный кэш**: Вся база домов и улиц (всего 70 строк) кэшируется на фронтенде в `Cabinet.tsx` один раз при загрузке страницы, что исключает сетевые задержки (debounce) при каждом нажатии клавиши и гарантирует моментальный отклик интерфейса (0 мс).

## Дата: 2026-05-24 (Итоговое решение миграции)
### Изменения:
- **Отказ от экосистемы Supabase**: Удален локальный `supabase start` по просьбе владельца, чтобы гарантировать 100% изоляцию и использование чистой базы PostgreSQL.
- **Внедрение локального API (PostgREST)**: В `docker-compose.yml` добавлен легковесный контейнер `postgrest`, который предоставляет REST API напрямую к нашей локальной базе PostgreSQL. Это позволило сохранить работоспособность **всех 23 файлов фронтенда**, использующих `supabase.from()`, без необходимости их переписывания.
- **Интеграция Auth**:
  - Фронтенд (`Auth.tsx`) использует наш кастомный `Node.js` бэкенд (`/api/auth/login`) для генерации JWT токенов.
  - В `server/index.js` обновлена генерация JWT — теперь токен содержит поле `role: 'authenticated'`, что позволяет PostgREST безопасно принимать запросы от авторизованных пользователей.
- **Polyfill для Supabase SDK**:
  - В `src/integrations/supabase/client.ts` переопределен клиент Supabase. Теперь все запросы направляются на `http://localhost:3000` (наш PostgREST) вместо облака.
  - В `client.ts` реализован перехватчик URL для обхода префикса `/rest/v1/`, что позволяет прозрачно работать с локальным PostgREST.
  - Написан polyfill для метода `supabase.auth.getSession()`, который читает `auth_token` из `localStorage`.

### Исправление проблемы с JWT и авторизацией:
- Устранена ошибка 401 "None of the keys was able to decode the JWT".
- Проблема заключалась в том, что `docker-compose restart` не применял новые переменные окружения к контейнеру PostgREST. Использование `docker-compose up -d` полностью пересоздало контейнер с новым общим ключом.
- Теперь бэкенд и PostgREST используют единый строковый ключ (`super-secret-jwt-token-with-at-least-32-characters-long`).

- `domofond_postgrest` (порт 3000) — Прокси для базы данных (заменяет функционал Supabase API).
